=======================================
Full canonical workspace (41 crates) + design notes + fresh firmware ELFs.

This archive: virtio-net TX/RX over the virtqueues + the #GP-fault fix.

  1. virtio-net TX/RX (kernel-image/src/arch/virtio_net.rs):
     - Full legacy split virtqueue: descriptor table, avail ring, used ring,
       10-byte virtio_net_hdr, RX pre-fill + buffer recycling, TX send, RX recv.
     - Built on FrameAllocator::allocate_contiguous (physically-contiguous DMA
       frames; identity-mapped phys==virt). Queues set up via QUEUE_SELECT/
       QUEUE_SIZE/QUEUE_PFN; DRIVER_OK asserted after RX fill.
     - VERIFIED real TX: a broadcast frame (payload CIBOS-HELLO!) was transmitted
       through the TX virtqueue and captured in a QEMU filter-dump pcap.

  2. #GP fault fix (VIRTIO-GP-FAULT-DIAGNOSIS.md): with a NIC present the kernel
     #GP'd (vector 13, err 0x17a = IDT gate 0x2F = IRQ15 spurious). Root cause: a
     spurious/unhandled 8259 slave IRQ on a vector with no IDT gate. Faithful
     two-part fix (no shortcut):
       a) DRIVER: VRING_AVAIL_F_NO_INTERRUPT on both queues' avail rings (the
          polled driver tells the device not to raise completion interrupts).
       b) KERNEL: real 8259 spurious-IRQ handling — asm stubs + handler + IDT
          gates for 0x27 (master IRQ7) and 0x2F (slave IRQ15); reads the PIC ISR
          and does the correct conditional EOI (no EOI for true master spurious;
          master-only EOI for slave spurious). Robust to spurious/unexpected PIC
          interrupts that real hardware genuinely produces.
     VERIFIED: NIC boot now reaches `boot complete` with TX working and NO fault;
     no-NIC boot unchanged.

Both drivers + fix target standardized interfaces (virtio-pci spec, 8259 PIC);
QEMU merely verifies them. No QEMU-conditional logic anywhere.

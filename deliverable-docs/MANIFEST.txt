== PERIPHERAL DTB DISCOVERY (this increment) ==
  The remaining genuine peripheral hardcode (aarch64 PL011 UART 0x09000000) is now
  runtime-discovered:
  - arch/aarch64.rs: UART0 is an AtomicUsize defaulting to the QEMU-virt address
    (earlycon, so the very first kprintln works before the DTB is parsed), with
    set_uart_base() to override it. putc reads the atomic.
  - boot.rs: dtb_device_base(prefix) parses the DTB; kernel_entry overrides the
    UART base from the DTB's pl011 node right after stashing the pointer, so later
    output and real boards use the discovered address.
  - cibos-dtb: real-DTB test finds_pl011_uart_base proves device_base(b"pl011")
    resolves to 0x09000000 in the REAL captured QEMU DTB (node pl011@9000000).
  - The two-stage (earlycon default -> DTB override) pattern is the correct
    bare-metal approach: you cannot read the DTB to find the UART without first
    having a UART to report DTB errors on. On QEMU `-kernel` (no DTB pointer) the
    default is used; the override engages when a real DTB arrives (CIBIOS/U-Boot).
  - riscv64 console uses OpenSBI (SBI calls) — no UART MMIO to discover; correctly
    left until a PLIC/CLINT driver lands. GIC/PLIC/CLINT are not yet driven by any
    kernel driver, so nothing is hardcoded-and-used there to fix today (flagged).
  See PERIPHERAL-DTB-DISCOVERY.md and DTB-AND-HARDCODED-VALUES.md.

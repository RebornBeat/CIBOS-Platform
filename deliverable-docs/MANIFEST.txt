== THE BRING-UP CONTRACT (this milestone) ==
The correction: x86_64 is not "one sample to generalize from later" — it is the
REFERENCE IMPLEMENTATION of the canonical HIP bring-up. So the contract was
extracted from x86_64 NOW and every other arch implements that SAME contract,
aligning all four by construction from the first line (rather than growing each
arch separately and reconciling drift later).

  - kernel-image/src/bringup.rs: trait ArchBringUp + PhaseStatus{Done,Skipped,
    Failed}. Phases (canonical order): early_traps -> seed_entropy ->
    mount_root_fs -> bring_up_mmu (owns frames; NIC + ring-3 nested in its scope
    on x86_64) -> verify_storage. Portable phases (heap, handoff, scheduler core)
    run directly in kernel_entry on every arch.
  - boot.rs: FOUR side-by-side `impl ArchBringUp for Arch` (x86_64/aarch64/
    riscv64/x86). x86_64 delegates to the existing verified functions (NO
    behavior change). aarch64/riscv64/x86 return Skipped("pending: <phase>") for
    unbuilt phases — honest, and filled in later by implementing the SAME method,
    never a new cfg block.
  - kernel_entry: drives the phases through `Arch` in canonical order with ZERO
    target_arch cfgs in the boot control flow (was 5+ scattered). The redundant
    non-x86 bring_up_mmu stub was deleted.

VERIFIED:
  - x86_64 full stack UNCHANGED: MMU online, container isolation, DNS STACK OK,
    REMOTE LINK OK, boot complete. No "skipped" lines (does every phase).
  - aarch64 boots core + reports its pending phases through the contract:
    "entropy seed skipped (pending: aarch64 RNG path)",
    "MMU bring-up skipped (pending: aarch64 MMU encoder (TTBR/4KB granule))".
  - riscv64 boots core + reports its pending phases (RNG, Sv39 MMU).
  The per-arch roadmap is now visible in the boot log itself; building an arch =
  replacing a Skipped phase with a real impl, x86_64's impl as the spec.

== UNDERLYING (prior milestones, still verified) ==
  A1 — remote Lattice Links over the NIC: a Link handle resolves to a local
  Channel OR a RemoteLink UDP flow; link_send/link_recv dispatch with ZERO
  surface/ABI change. DNS round-trip via the real Link API: REMOTE LINK OK.
  e1000 MMIO fix: identity-map the PCI MMIO hole (0xFEB00000..0xFEC00000) so the
  e1000 BAR0 is reachable. All three NIC paths work (none / virtio-net / e1000).
  aarch64 boot: FP/SIMD enable (CPACR_EL1) + exception vectors (VBAR_EL1).
  riscv64 boot: trap vector (stvec); needs OpenSBI (default -bios).

== NO-DRIFT / ALIGNMENT ==
  cibos-kernel (canonical core) has ZERO target_arch refs — every arch runs the
  IDENTICAL core. The bring-up contract now extends that guarantee to the
  arch-specific layer: identical phase sequence, per-arch impls auditable as one.
  Docs: ARCH-BRINGUP-CONTRACT-DESIGN.md, ARCH-ABSTRACTION-AND-ALIGNMENT.md,
  AARCH64-BRINGUP-DIAGNOSIS.md, REMAINING-ARC-AND-ARCH-WIFI-PLAN.md.

== Bare-metal-first ==
  Every backend targets STANDARD interfaces unaware of QEMU (PL011, SBI, 8259
  PIC, PCI, e1000 MMIO, VBAR_EL1, stvec, CPACR_EL1). QEMU VERIFIES; we never
  build "for QEMU".

== NETWORKING (x86_64) ==
  A1 — remote Lattice Links over the NIC (VERIFIED): a Link handle now resolves
  to EITHER a local Channel (loopback) OR a RemoteLink UDP flow, and link_send/
  link_recv dispatch on which — with ZERO change to the Gate/Link/Warden surface
  or the syscall ABI. A remote Link round-trips a DNS query through the real
  link_send/link_recv API over the NIC:
      lattice remote Link: reply via link_recv (52 bytes) — REMOTE LINK OK
  This realizes NETWORKING.md's promise: only a Link's backing transport widens.

  e1000 MMIO-hole fix (VERIFIED): the Intel 82540EM driver hung because its MMIO
  BAR0 (0xFEB80000) sat OUTSIDE the kernel's 1 GiB identity map. Fix: identity-map
  the PCI MMIO hole (0xFEB00000..0xFEC00000). A latent N3 bug (not an A1
  regression), found via QEMU instruction tracing. All three NIC paths now work:
  -nic none (clean), virtio-net (full A1 stack), e1000 (real EEPROM MAC + TX/RX).

== PER-ARCH SWEEP (the big alignment question) ==
  Honest four-arch state:
    x86_64  — FULL stack (ring-3, all drivers, networking, Lattice). The only
              complete arch; re-verified this session (DNS STACK OK + REMOTE LINK
              OK + boot complete, no regression from multi-arch work).
    aarch64 — core BOOTS. Root-caused the silent death via QEMU -d in_asm,int:
              FP/SIMD was trapped (CPACR_EL1.FPEN) and Rust core uses SIMD ->
              EC 0x7. Fixes: enable FP/SIMD in boot/aarch64.s; install exception
              vectors (vectors_aarch64.s + VBAR_EL1 + ESR/ELR/FAR reporter). Now
              runs heap->handoff->scheduler->init lane->boot complete.
    riscv64 — core BOOTS. Was NOT broken; the earlier "silent" run used
              `-bios none` (no OpenSBI console) — a test-harness error, not a
              kernel bug. Added a trap vector (vectors_riscv64.s + stvec +
              scause/sepc/stval reporter) for fault visibility parity. Boots to
              boot complete under default OpenSBI.
    i686    — builds (firmware + kernel ELF) but the backend (x86.rs) is a
              COM1 serial + halt STUB, NOT the full x86_64 stack. Boots via the
              CIBIOS firmware path (cannot -kernel boot: no PVH note). Full i686
              runtime parity (32-bit IDT/paging + reuse x86 drivers) is a later
              increment. So "x86 legacy" does NOT fully build the stack yet.

== NO-DRIFT / ALIGNMENT ASSESSMENT (ARCH-ABSTRACTION-AND-ALIGNMENT.md) ==
  - cibos-kernel (canonical core) has ZERO target_arch references — the HIP
    invariants are arch-free, so every arch runs the IDENTICAL core. aarch64 +
    riscv64 ran the scheduler/init lane with NO core changes: the no-drift
    guarantee by construction.
  - The arch contract (arch/mod.rs) is clean: all arches export halt/init_serial/
    putc; x86_64 additionally exports its full set; others grow toward parity.
    Per-arch branching is isolated to boot.rs (28/33 cfgs x86_64-gated, cohesive).
  - Guardrail: extract a per-arch BringUp contract when the SECOND arch needs the
    same timer/MMU/ring-3 phases — not now (premature abstraction avoided).

== Bare-metal-first (unchanged) ==
  Every backend targets STANDARD interfaces, unaware of QEMU: x86 (VGA 0xB8000,
  COM1 0x3F8, 8259 PIC, PCI 0xCF8, e1000 MMIO), aarch64 (PL011 @ 0x09000000,
  VBAR_EL1, CPACR_EL1), riscv64 (SBI console, stvec). QEMU presents these standard
  interfaces, so it VERIFIES the code; we never build "for QEMU". A device tree
  was used ONLY to confirm the PL011 address is the standard one.

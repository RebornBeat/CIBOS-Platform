== MMU: one orchestration, three arches (this milestone) ==
The earlier concern that bring_up_mmu's coupling forced per-arch reimplementation
was wrong: the coupling was ORCHESTRATION, not arch. The arch-specific surface was
tiny and already half-abstracted (cibos_kernel::paging::PageTableEncoder). So:
  - bring_up_mmu_generic<P: ArchPaging> = ONE portable orchestration (collect
    regions, frame allocator, build tables, identity-map, install, verify
    isolation, post-MMU phases). kernel-image/src/bringup.rs defines ArchPaging.
  - Per-arch hooks (~7 small items): type Encoder, identity_map_bytes,
    reserved_below, mmio_identity_ranges, enable_table_features, install,
    current_root.
  - x86_64 (arch/paging.rs): X86PageTable + CR3 + EFER.NXE + PCI MMIO hole.
    UNCHANGED — full stack re-verified (MMU online, container isolation, DNS
    STACK OK, REMOTE LINK OK, boot complete).
  - aarch64 (arch/paging_aarch64.rs): VMSAv8-64 4KB encoder + TTBR0_EL1/TCR/MAIR/
    SCTLR. "MMU online — root 0x42000000", boot complete.
  - riscv64 (arch/paging_riscv64.rs): Sv48 encoder (Sv48 is 4-level, matching the
    portable geometry; Sv39 is 3-level and would not) + satp. "MMU online — root
    0x82000000", boot complete. Worked FIRST TRY.
All three build their own hardware page tables through the SAME code and keep
executing past the install — the unfakeable proof the tables are valid.

The arch differences that surfaced were all DATA, not code: RAM base (x86 1 MiB /
aarch64 1 GiB / riscv64 2 GiB — READ FROM EACH BOARD'S DTB, not guessed),
reservation watermark, and using Sv48 to match the page-walk geometry. The
no-drift design proved itself: one path, per-arch data.

== HONEST VERIFICATION REALITY (VERIFICATION-REALITY-AND-BARE-METAL.md) ==
Important and not glossed over:
  - This sandbox is x86_64-only, NO KVM, NO real aarch64/riscv64 hardware, NO
    dual-boot. EVERY arch (incl. x86_64) is QEMU EMULATION, not real silicon.
  - "Verified" = ran correctly against QEMU's faithful ISA MODEL. The
    architecture-level work (encoders, MMU enable, traps) targets the ISA spec and
    largely TRANSFERS to real hardware. The PLATFORM specifics (RAM base, device
    addresses) are currently QEMU-virt constants — correct for QEMU virt (checked
    against each DTB) but WRONG for other boards.
  - The bare-metal-correct fix (OUTSTANDING): parse the DTB the firmware passes at
    runtime (currently ignored) so the kernel reads the real layout and works on
    QEMU AND real hardware without knowing which. Then real-hardware testing
    OUTSIDE this sandbox. Neither is done; no real-silicon claim is made.

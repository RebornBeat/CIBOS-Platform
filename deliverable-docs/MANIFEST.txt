== Dwell: DTB walker hardening (real bare-metal crash fix) ==
  A truncated/corrupt DTB made prop_name index past the strings block and PANIC —
  malformed firmware data would crash the kernel during device discovery. Fixed
  prop_name to bounds-check (empty name on overflow). Audited all walker slicing:
  node-name slices are loop-bounded; be32/be64 + reg-data use .get() (Option).
  Added a truncated-blob test over count_nodes/device_reg_lowest/find_prop_u32/
  node_has_prop — all safe. This is hardening QEMU's always-well-formed DTB never
  exercised.

== CIBOSFS on virtio-blk (aarch64/riscv64 root filesystem) ==
  The full filesystem stack now runs on virtio-blk — the same arch-independent
  Fs<D: BlockDevice> layer x86 uses on ATA, only the device differs. After the
  block-layer LBA 0 check the kernel formats CIBOSFS, mkdir(/etc), writes
  /etc/hello, reads it back, then REMOUNTS and re-reads to prove persistence to the
  medium (not RAM).
  VERIFIED: aarch64 (slot 31) and riscv64 (slot 7) both print "CIBOSFS on
  virtio-blk — format/write/read-back/remount OK". This also exercises the riscv64
  Zicbom cache-maintenance path under real multi-sector filesystem I/O. No-disk
  degrades gracefully; x86 ATA unaffected; 383 tests pass.
  HONEST SCOPE: proves the fs-on-virtio-blk path end to end. Installing it as the
  persistent kernel ROOT_FS so the filesystem SYSCALLS operate on it
  (aarch64/riscv64) is the next step (needs a per-arch ROOT_FS<Fs<VirtioBlkMmio>> or
  a dyn-BlockDevice refactor); deferred to keep this increment focused + verified.
